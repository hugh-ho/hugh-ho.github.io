最近，公司内部正在大力推广 AI Coding。Prompt、Agent、Harness、Loop，一波又一波的新概念不断出现，模型的编码能力也在快速提升。

但实践一段时间后，我越来越强烈地感受到：**AI Coding 在大厂真正的瓶颈，已经不只是模型能不能写出代码，而是公司的工程基础设施能不能跟上。**

模型可以在几分钟内完成一轮代码修改，但要验证这次修改，仍可能需要申请环境、部署服务、准备数据、等待流水线，再由人去查看日志和判断结果。代码生成快了，反馈链路却没有变短，最终的研发效率自然不会出现数量级的提升。

换句话说，我们不应该只关注 Agent 一次生成了多少代码，更应该关注：**从需求提出，到产生一个经过验证、可以交付的变更，究竟需要多长时间。**

## 真正限制 Agent 的，是反馈速度

过去，开发完成后将代码部署到预发环境，再进行联调和测试，是一套可以接受的流程。但到了 AI Coding 场景，这套流程会成为明显的串行瓶颈。

Agent 每完成一轮修改，都需要从真实环境中获得反馈：代码能否编译、服务能否启动、接口行为是否符合预期、日志里有没有异常、性能指标是否发生退化。只有拿到这些反馈，它才能继续修正代码，形成闭环。

Anthropic 在 [Building Effective Agents](https://www.anthropic.com/engineering/building-effective-agents) 中提到，Agent 在执行过程中需要不断从环境中获得“ground truth”，再根据工具调用和代码执行的结果判断进展。对于自主程度更高的 Agent，还需要在沙箱环境中进行充分测试，并设置必要的边界和保护机制。

因此，如果每次验证都必须经历一次远程部署，Agent Loop 的速度就会被部署链路锁死。模型推理只需要几分钟，等待环境却可能需要几十分钟。再强的模型，也只能在漫长的等待中原地踏步。

我们目前探索的一种方案，是利用网关或服务发现能力做流量路由：把原本发往测试环境某个服务的请求，转发到开发机或隔离环境中启动的本地服务；对于暂时无法接入的下游服务，则通过 Mock 提供稳定、可重复的数据。这样，Agent 修改代码后就能立即启动服务、发起请求、检查结果，而不必等待完整部署。

这类能力的价值并不只在于“少部署一次”，而是把原来依赖人工操作的测试过程，变成 Agent 可以自主执行的反馈回路。

## 企业真正需要建设的四层能力

我理解的 Harness，不只是包裹模型的一段 Agent Loop，而是一整套让 Agent **看得见、做得到、验得过、交得出**的工程系统。要让 AI Coding 真正进入日常研发流程，至少需要补齐以下四层能力。

### 1. 可被 Agent 理解的知识底座

知识库是 Coding Agent 的底座，但它不应该只是把文档切片后塞进向量数据库。Agent 真正需要的是一张可以逐层展开的工程地图，包括：

- 系统架构、模块边界和依赖关系；
- 编码规范、技术约束和常见设计模式；
- 本地启动、测试、发布和故障排查方式；
- 业务术语、历史决策以及关键链路说明。

我们可以通过 Code4Context 一类能力帮助 Agent 检索代码和文档，并使用 Skill 把这些知识接入开发流程。但“能搜到”还不够，还需要持续评测它是否“找得对、用得上”。例如，通过 GitLab App 获取真实研发任务和变更，由后台评估 Agent 检查上下文召回是否准确、生成结果是否遵守约束，再把失败案例沉淀成新的评测集。

OpenAI 在 [Harness engineering](https://openai.com/index/harness-engineering/) 的实践中，也把仓库内的知识视为系统事实来源：入口文档只负责提供地图，更详细的架构、设计和执行计划则分层组织在仓库中。这个思路很重要——**给 Agent 一张地图，而不是一本塞满所有细节的千页手册。**

### 2. 面向内部平台的工具与 Skill

模型知道应该做什么，不代表它有能力完成。企业内部大量能力散落在代码平台、云平台、发布系统、日志平台、监控平台和权限系统中。如果这些系统只能由人点击页面操作，对 Agent 来说，它们就等于不存在。

需要通过稳定的 CLI、API 或 Skill，把关键研发动作变成 Agent 可以调用的工具，例如：

- 查询代码、Issue 和历史变更；
- 创建隔离环境并启动服务；
- 执行构建、发布和回滚；
- 查询日志、指标和链路追踪；
- 获取评审意见并继续修改。

这些工具不仅要“能调用”，还要有清晰的输入输出、可判断的错误信息、合理的权限边界和幂等能力。否则，Agent 只会把原来的人肉操作，变成更加难以排查的自动化操作。

### 3. Agent 可以自主运行的测试环境

测试是当前最值得投入、也最容易形成杠杆的环节。理想状态下，每个任务都有独立的工作区和运行环境，Agent 可以在其中启动应用、准备测试数据、调用接口，并读取日志、指标和测试报告。

除了单元测试和静态检查，还需要逐步补齐：

- 基于流量路由的本地联调能力；
- 可复现的下游服务 Mock；
- 接口、UI 和端到端自动化测试；
- Agent 可直接查询的日志、指标与 Trace；
- 与生产环境隔离的沙箱和安全护栏。

OpenAI 的实践中，会为每个 Git worktree 启动独立应用实例，并把 UI、日志和指标暴露给 Agent，使它能够复现问题、验证修复结果。这里最值得借鉴的不是某个具体工具，而是一个原则：**任何 Agent 无法观察的系统状态，都无法进入它的纠错闭环。**

### 4. 一份可执行、可验证的交付标准

Agent Loop 不能只靠一句“继续优化，直到完成”。什么叫完成，必须被定义成机器可以检查的标准。

一份任务交付标准至少应该包括：

1. 需求和边界是否清晰；
2. 方案是否符合现有架构约束；
3. 代码是否通过编译、Lint 和测试；
4. 核心场景是否完成验证；
5. 日志、指标和安全风险是否检查；
6. 变更说明、回滚方案和评审材料是否齐全。

有了明确的验收条件，Agent 才能按照“设计—编码—测试—评估—纠错”的过程不断循环，并在满足条件时停止，而不是生成一段看起来正确的代码就宣告完成。

## 衡量提效，也需要换一把尺子

如果仍然用代码补全率、生成代码行数或 Agent 调用次数衡量 AI Coding，很容易得到虚假的繁荣。真正有意义的指标应该更接近最终交付，例如：

- 从任务开始到首次验证通过的时间；
- Agent 独立完成任务的比例；
- 每个任务需要人工接管的次数；
- 变更进入测试环境或生产环境后的回归率；
- 完成一次有效交付所消耗的模型与工具成本。

这些指标会迫使我们把注意力从模型演示转向工程闭环：到底是知识缺失、工具不可用、环境太慢，还是验收标准不清楚？每一次 Agent 失败，都应该反过来推动文档、工具、测试或护栏的改进。

## 写在最后

AI Coding 的上半场，大家在比较谁的模型写代码更快、更准；到了下半场，竞争的重点会逐渐变成：谁能为 Agent 提供更完整的上下文、更可靠的工具、更快速的测试环境，以及更清晰的交付标准。

模型能力还会继续快速发展，但企业内部的工程基础设施不会自动完成升级。只有把方案设计、编码、测试、评审和纠错串成一个低延迟、可观测、可验证的循环，模型能力才能真正转化为研发效能。

目前的实践距离“大幅提效”还有不小的距离。但方向已经越来越清楚：**不要只给 Agent 一个更聪明的大脑，还要为它建设一套能够独立工作的工程环境。**
