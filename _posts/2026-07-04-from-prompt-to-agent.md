## 引言：我们为什么需要 Agent？

大语言模型（LLM）已经展现出惊人的理解和生成能力，但它本质上仍是一个**静态的知识库**——只能基于训练数据回答问题，无法获取实时信息、执行实际操作或进行多步骤规划。

**Agent（智能体）** 的核心思想是：给 LLM 配备"手脚"和"大脑"，让它能够：
- 🧠 **思考**：分解复杂问题，制定执行计划
- 🔧 **行动**：调用工具获取信息、执行代码、操作外部系统
- 💾 **记忆**：保存和检索知识，支持长期对话
- 🔄 **反思**：评估执行结果，自我纠错和改进

接下来，我们将从**提示工程**这个基础开始，一步步构建完整的 Agent 能力。

## 提示工程——Agent 的"语法基础"

### Zero-shot、One-shot 与 Few-shot

在让模型做任务前，我们需要知道如何**描述任务**。根据是否提供示例，有三种方式：

**Zero-shot（零样本）**：直接描述任务，不给例子
```
请将以下中文翻译成英文："今天天气很好"
```

**One-shot（单样本）**：给一个例子，让模型"照葫芦画瓢"
```
把中文翻译成英文：
中文：你好 → 英文：Hello
中文：今天天气很好 → 英文：
```

**Few-shot（少样本）**：给 2-5 个例子，让模型学习规律
```
判断情感倾向：
句子：这部电影太棒了！ → 情感：正面
句子：服务态度很差 → 情感：负面
句子：这个还可以吧 → 情感：
```

**核心洞察**：示例可以帮助模型理解**输出格式、风格偏好、任务边界**，这是后续 Agent 设计的基础。

### 思维链（Chain-of-Thought, CoT）

对于复杂问题，直接要答案往往出错。CoT 的核心是：**让模型"一步一步思考"，显式写出推理过程**。

**对比示例**：

❌ 无 CoT：
```
问：一个农场有鸡兔共35只，脚94只，各几只？
答：23只鸡，12只兔（可能算错）
```

✅ 有 CoT：
```
问：鸡兔同笼，35头94脚，各几只？

答：让我逐步分析：
1. 假设全是鸡，应有 35×2=70 只脚
2. 实际多了 94-70=24 只脚
3. 每只兔多2只脚，所以兔有 24÷2=12 只
4. 鸡有 35-12=23 只
答案：23只鸡，12只兔
```

**进阶技巧**：只需在 Prompt 末尾加一句"**Let's think step by step**"，模型就能自动展开推理。

### 深入理解 Temperature：控制随机性的数学原理

**Temperature（温度）** 是控制 LLM 输出随机性的核心参数。它不是一个简单的"开关"，而是通过数学变换调整概率分布的形状。

#### 直观理解

| Temperature | 效果 | 适用场景 |
|-------------|------|----------|
| **0.0-0.3** | 确定性强，几乎总是选概率最高的词 | 代码生成、数学计算、事实问答 |
| **0.7-1.0** | 平衡，有一定多样性但不离谱 | 通用对话、写作辅助 |
| **1.2-1.5** | 高随机性，创意发散 | 头脑风暴、诗歌创作、故事生成 |

#### 数学原理：Temperature 如何改变概率

LLM 输出 logits（原始分数）后，通过 Softmax 转换为概率。Temperature 在这一步起作用：

**步骤 1：Temperature 缩放**
```
z_i' = z_i / T
```
将每个 logit 除以 Temperature T。

**步骤 2：Softmax 归一化**
```
P_i = exp(z_i') / Σ exp(z_j')
```

**具体示例**：假设模型对三个候选词的 logits 为 `[2.0, 1.0, 0.5]`

| T | 计算过程 | 概率分布 | 效果 |
|---|---------|---------|------|
| **0.5** | 除以 0.5 → `[4.0, 2.0, 1.0]` → exp | `[0.84, 0.11, 0.04]` | 差距放大，强者愈强 |
| **1.0** | 不变 → `[2.0, 1.0, 0.5]` → exp | `[0.63, 0.23, 0.14]` | 原始分布 |
| **2.0** | 除以 2 → `[1.0, 0.5, 0.25]` → exp | `[0.48, 0.29, 0.23]` | 差距缩小，更均匀 |

**关键洞察**：
- **T < 1**：指数函数 `exp(x/T)` 放大了 logits 之间的差距 → 分布更"尖锐"
- **T > 1**：缩小了 logits 之间的差距 → 分布更"平坦"
- **T → 0**：最大 logit 对应的概率趋近于 1，其他趋近于 0（one-hot 分布）
- **T → ∞**：所有概率趋近于相等（均匀分布）

#### 为什么 T=0 时几乎总是选概率最高的词

当 T 趋近于 0 时，指数函数的"爆炸性增长"会把 logits 之间微小的差距，放大成天文数字级别的差距：

```
T = 0.1 时：
- 最大 logit 2.0 → exp(2.0/0.1) = exp(20) ≈ 4.85 × 10^8
- 次大 logit 1.0 → exp(1.0/0.1) = exp(10) ≈ 2.20 × 10^4
- 差距：约 22,000 倍

T = 0.01 时：
- exp(200) vs exp(100) → 差距约 e^100 ≈ 10^43 倍！
```

**极限情况（T → 0）**：
- 最大 logit 的 exp(z_max/T) → **无穷大**
- 其他所有 exp(z_j/T) 相对它都是 **0**

因此，概率分布变成：**最大概率 ≈ 1，其他 ≈ 0**，模型实际上总是选择概率最高的词（贪婪解码）。

#### Temperature vs 采样策略：两个独立的概念

**重要区分**：Temperature 调整概率分布的**形状**，而**采样策略**决定如何根据分布选择词。

| 策略 | 说明 | Temperature 的作用 |
|------|------|-------------------|
| **贪婪解码（Greedy）** | 总是选概率最高的词 | T 几乎不影响输出（T=0 时完全确定） |
| **随机采样（Sampling）** | 按概率分布随机抽取 | T 决定"抽取时各候选概率差距多大" |

**对比示例**（使用随机采样，抽 100 次）：

| T | 天空(50%) 大海(30%) 梦想(15%) 喵呜(5%) | 结果 |
|---|----------------------------------------|------|
| **0.1** | 概率变成 `[0.99, 0.009, 0.001, ~0]` | 天空≈99次，其他极少 |
| **1.0** | 保持 `[0.50, 0.30, 0.15, 0.05]` | 接近原始比例 |
| **2.0** | 变成 `[0.35, 0.30, 0.20, 0.15]` | 低概率词更有机会 |

#### 实际 API 中的行为

主流 API 的默认行为：

```python
# OpenAI/Claude 示例

# 贪婪解码（确定性输出）
response = client.chat.completions.create(
    model="gpt-4",
    messages=[{"role": "user", "content": "计算 2+3*4"}],
    temperature=0  # 或 top_p=0，强制贪婪解码
)
# 结果：每次调用返回相同答案 "14"

# 随机采样（有创造性）
response = client.chat.completions.create(
    model="gpt-4",
    messages=[{"role": "user", "content": "写一首关于月亮的诗"}],
    temperature=0.8  # 有随机性，每次可能不同
)
# 结果：每次调用可能生成不同的诗
```

**注意**：`temperature=0` 时，由于 GPU 浮点精度问题，理论上应该有极微小的不确定性，但主流平台会为 `T=0` 走专门的贪婪解码路径，确保输出确定性。

#### 其他相关参数

Temperature 通常与以下参数配合使用：

| 参数 | 作用 | 典型值 |
|------|------|--------|
| `top_p` (nucleus) | 只从累积概率前 p% 的词中采样 | 0.1-1.0 |
| `top_k` | 只从概率最高的 k 个词中采样 | 1-100 |
| `presence_penalty` | 降低已出现词的概率 | 0-2 |
| `frequency_penalty` | 降低高频重复词的惩罚 | 0-2 |

**常见组合**：
- **精确模式**：`T=0` + 贪婪解码
- **平衡模式**：`T=0.7, top_p=1.0`（默认）
- **创意模式**：`T=1.2, top_p=0.9`
- **质量控制**：`T=0.8, top_k=40`（限制候选池，避免太差的选择）

## 从提示到 Agent——为什么需要 Agent？

提示工程让 LLM 变得更"聪明"，但仍有两个根本局限：

1. **知识截止**：无法获取训练日期后的信息
2. **无法行动**：不能执行代码、查询数据库、调用 API

**ReAct（Reasoning + Acting）** 范式解决了这个问题——让模型交替进行**思考**和**行动**：

```
思考：用户问北京今天天气，我需要查询实时信息
行动：调用天气工具(city="北京")
观察：{温度: 25, 天气: 晴}
思考：已获得信息，可以回答用户
回答：北京今天晴天，25度
```

这就是 Agent 的核心：**LLM 作为"大脑"，负责决策；工具作为"手脚"，负责执行**。

## Agent 的推理方法——从链到图

随着任务复杂度提升，单一的线性推理（CoT）不够用了。以下是主流的多步骤推理方法：

### ReAct：推理与行动的循环

**ReAct（Reasoning + Acting）** 是 Agent 架构的基石，由 Google 在 2022 年提出。它通过让 LLM **交替进行推理和行动**，解决了纯文本推理与外部世界交互的问题。

#### 为什么需要 ReAct？

对比三种范式：

| 范式 | 机制 | 局限性 |
|------|------|--------|
| **仅 Action** | 直接调用工具 | 没有推理过程，工具调用可能是错误的 |
| **仅 CoT** | 仅内部思考 | 无法获取外部信息，容易"幻觉" |
| **ReAct** | 思考 → 行动 → 观察 → 循环 | 既有推理指导行动，又能通过观察更新推理 |

**核心洞察**：人类解决问题时，也是**思考-行动-观察**的循环过程。ReAct 模拟了这一认知模式。

#### ReAct 的三个核心组件

```
┌─────────────────────────────────────────────────────────────┐
│                        ReAct 循环                            │
│                                                             │
│  ┌──────────┐      ┌──────────┐      ┌──────────┐          │
│  │ Thought  │─────→│  Action  │─────→│Observation│          │
│  │  思考    │      │  行动    │      │  观察    │          │
│  │          │      │          │      │          │          │
│  │ "我需要  │      │ 调用工具 │      │ 获取结果 │          │
│  │  搜索"   │      │ search() │      │ "结果是xx"│          │
│  └──────────┘      └──────────┘      └──────────┘          │
│        ↑                                    │               │
│        └────────────────────────────────────┘               │
│                      (循环直到完成)                          │
└─────────────────────────────────────────────────────────────┘
```

**1. Thought（思考）**
- 分析当前状态和目标
- 决定下一步需要什么信息
- 评估已有信息是否足够

**2. Action（行动）**
- 根据思考结果调用工具
- 格式通常是标准化的（如 JSON）
- 行动类型：搜索、计算、API 调用、数据库查询等

**3. Observation（观察）**
- 接收工具返回的结果
- 将结果纳入上下文
- 为下一轮思考提供输入

#### 完整工作流程示例

**场景**：用户询问"苹果公司最新股价是多少？比一年前涨了多少？"

```
用户：苹果公司最新股价是多少？比一年前涨了多少？

═══════════════════════════════════════════════════
第 1 轮
───────────────────────────────────────────────────
思考1：用户询问苹果公司的股价信息，包括当前价格和一年前的对比。
      我需要先获取苹果公司的股票代码，然后查询当前股价和一年前的股价。
      苹果公司的股票代码是 AAPL。

行动1：get_stock_price(symbol="AAPL")

观察1：{"current_price": 185.50, "currency": "USD", "date": "2024-07-04"}

═══════════════════════════════════════════════════
第 2 轮
───────────────────────────────────────────────────
思考2：已获得当前股价 $185.50。现在我需要获取一年前的股价来计算涨跌幅。
      一年前的日期大约是 2023-07-04。

行动2：get_historical_price(symbol="AAPL", date="2023-07-04")

观察2：{"price": 173.50, "date": "2023-07-04"}

═══════════════════════════════════════════════════
第 3 轮
───────────────────────────────────────────────────
思考3：现在我有了两个数据点：
      - 当前股价：$185.50
      - 一年前股价：$173.50
      涨幅 = (185.50 - 173.50) / 173.50 × 100% = 6.92%
      我已经有足够的信息回答用户了。

行动3：finish

最终答案：苹果公司(AAPL)最新股价为 $185.50。
         相比一年前($173.50)，上涨了约 6.92%。
```

#### ReAct vs 纯 CoT 的对比

**同样的问题：苹果公司股价涨跌幅**

❌ **纯 CoT（容易幻觉）**：
```
问：苹果股价比一年前涨了多少？

答：让我计算一下...
苹果当前股价大约是 $180（模型猜测，可能不准确）
一年前大约是 $150（模型猜测，可能不准确）
涨幅 = (180-150)/150 = 20%
```
**问题**：模型在"瞎猜"数字，结果不可信。

✅ **ReAct（准确可靠）**：
```
思考：我需要查询实时股价数据
行动：调用股价查询工具
观察：获得真实数据
思考：基于真实数据计算
答案：准确的涨跌幅
```

#### ReAct 的 Prompt 设计

为了让 LLM 遵循 ReAct 范式，需要在 System Prompt 中明确定义格式：

```
你是一个智能助手，可以调用工具解决问题。

每次回复必须遵循以下格式：
思考：[分析当前情况，决定下一步行动]
行动：[工具名称](参数)

当获得观察结果后，继续思考并决定下一步。
当任务完成时，输出：
行动：finish
答案：[最终回答]

可用工具：
- search(query): 搜索网络信息
- calculate(expression): 执行数学计算
- get_weather(city): 获取天气信息
```

**关键设计要点**：
1. **强制格式**：用换行符和标签确保结构化输出
2. **停止词**：在 Action 后停止生成，等待外部执行
3. **错误处理**：定义工具调用失败时的回退策略

#### 错误处理与边界情况

ReAct 循环中可能遇到的问题：

| 问题 | 解决方案 |
|------|----------|
| **工具调用失败** | Observation 返回错误信息，Thought 重新评估策略 |
| **信息不足** | Thought 决定需要更多搜索或换个角度 |
| **无限循环** | 设置最大轮数限制（如 max_iterations=10） |
| **工具不存在** | Thought 重新选择可用工具 |
| **格式错误** | 重试机制或提示修正 |

**示例：工具调用失败的处理**
```
思考：我需要查询北京天气
行动：get_weather(city="北京")

观察：Error: API rate limit exceeded（API 限流）

思考：天气 API 限流了，我可以尝试使用备用数据源，
      或者告诉用户暂时无法获取天气信息。
行动：search(query="北京今天天气")

观察：[搜索结果包含天气信息]

思考：从搜索结果中找到了天气信息，可以回答了。
```

#### ReAct 的变体与扩展

| 变体 | 特点 | 场景 |
|------|------|------|
| **Base ReAct** | Thought → Action → Observation 循环 | 通用任务 |
| **ReAct + CoT** | 在 Thought 中强制详细推理 | 需要深度分析的任务 |
| **ReAct + Reflexion** | 失败后反思并重试 | 容易出错的任务 |
| **Multi-turn ReAct** | 支持用户中途干预 | 交互式应用 |

ReAct 是所有现代 Agent 框架（LangChain、AutoGen 等）的基础构建块。理解 ReAct，就理解了 Agent 如何"思考"和"行动"的核心机制。

### 3.2 ToT（Tree of Thoughts）：思维树

当一个问题有多种解决思路时，CoT 的单一路径可能陷入局部最优。ToT 采用**树形结构**，同时探索多条路径：

```
        初始问题
       /   |   \
     思路A 思路B 思路C
    /  |      |
   A1  A2    B1
      /
    A2-1  ← 评估后选最优
```

**关键步骤**：
1. **生成**：对每个节点生成多个候选思路
2. **评估**：LLM 自我评估各候选的得分
3. **选择**：用 BFS 或 DFS 搜索最优路径

**适用**：创意写作、谜题解答、需要探索多种方案的场景。

### 3.3 GoT（Graph of Thoughts）：思维图

ToT 的树结构有个限制：不同分支无法合并。GoT 引入**图结构**，支持更灵活的操作：

- **聚合**：合并多个子思路的优点
- **精炼**：基于反馈改进已有思路
- **生成**：创建新节点

```
     思路A ────┐
         \      \
          ├→ 聚合思路D ─→ 最终答案
         /      /
     思路B ────┘
        ↑
     思路C（反馈循环）
```

### 3.4 Plan-and-Solve：先规划后执行

对于复杂的多步骤任务，把**规划**和**执行**解耦：

```
┌─────────────┐     ┌─────────────┐
│   Planner   │────→│  Executor   │
│  制定计划    │     │  执行每一步  │
│             │     │             │
│ 1. 查天气    │     │ → 调用API   │
│ 2. 查交通    │     │ → 查路线    │
│ 3. 给建议    │     │ → 生成回复  │
└─────────────┘     └─────────────┘
```

**优势**：规划阶段可以全局优化，执行阶段可以局部调整。

### 3.5 Reflexion：自我反思与迭代

人会在犯错后反思改进，Agent 也可以：

```
尝试1：解决问题 → 失败
   ↓
反思：分析失败原因（"第三步公式用错了"）
   ↓
尝试2：带着反思经验重试 → 成功
```

**核心组件**：
- **Evaluator**：评估结果质量
- **Self-Reflection**：生成改进建议
- **Memory**：存储失败经验

## 第四章：Agent 整体架构——所有组件如何协作

了解了各种推理方法（CoT、ReAct、ToT 等）之后，本节将展示**完整 Agent 的架构全景**。你会看到 Temperature、Memory、Tools 等各个模块是如何在 ReAct 框架下协同工作的。

### 4.1 整体架构图

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                              📥 用户输入层                                    │
│  "帮我查一下北京明天天气，然后推荐一个适合户外活动的地点"                       │
└─────────────────────────────────────────────────────────────────────────────┘
                                      ↓
┌─────────────────────────────────────────────────────────────────────────────┐
│                           🎯 规划层 (Planning)                               │
│  ┌──────────────────────────────────────────────────────────────────────┐  │
│  │ 任务分解：                                                           │  │
│  │ • 子任务1: 查询北京明天天气                                          │  │
│  │ • 子任务2: 根据天气推荐户外活动地点                                   │  │
│  │                                                                     │  │
│  │ 选择推理策略：                                                        │  │
│  │ • 简单任务 → CoT (单路径推理)                                        │  │
│  │ • 多方案比较 → ToT (树形探索)                                        │  │
│  │ • 复杂依赖 → GoT (图结构聚合)                                        │  │
│  │ • 需要验证 → Plan-and-Solve + Reflexion                             │  │
│  └──────────────────────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────────────────┘
                                      ↓
┌─────────────────────────────────────────────────────────────────────────────┐
│                        🧠 推理与决策层 (Reasoning)                           │
│                                                                             │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  ReAct 循环（核心引擎）                                              │   │
│  │                                                                     │   │
│  │   Thought ──→ Action ──→ Observation ──→ Thought ──→ ...           │   │
│  │      │          │            │                                     │   │
│  │      │          │            └─ 工具返回结果                        │   │
│  │      │          │                                                 │   │

│  │      │                                                            │   │
│  │      └─ "我需要查天气" / "基于晴天推荐公园"                         │   │
│  │                                                                     │   │
│  │  ┌─────────────────────────────────────────────────────────────┐   │   │
│  │  │  深度推理增强（可选）                                         │   │   │
│  │  │  • 复杂步骤使用 CoT: "让我逐步分析..."                         │   │   │

│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │

│  • top_p=0.9, max_tokens=500                                               │
└─────────────────────────────────────────────────────────────────────────────┘
                                      ↓
┌─────────────────────────────────────────────────────────────────────────────┐
│                         💾 记忆层 (Memory)                                   │

│   ┌──────────────┐    ┌──────────────┐    ┌──────────────┐                 │

│   │ 对话历史     │◄──►│ 当前任务状态  │◄──►│ 向量数据库    │                 │

```

### 4.2 组件间的数据流动

让我们追踪一个**完整请求**的数据流：

```
用户："苹果股价多少？"

  ↓
【规划层】
  • 识别为"查询类"任务
  • 选择策略：ReAct + 简单 CoT
  • Temperature=0.2（需要准确数据）

  ↓
【第1轮 ReAct】
  Thought: "需要查询苹果股票代码和当前价格"
  Action: search("AAPL stock price today")
  
  ↓
【工具层】
  • 调用搜索工具
  • 返回: {"AAPL": "$185.50"}

  ↓
【观察】
  Observation: "苹果当前股价 $185.50"

  ↓
【记忆更新】
  • 短期记忆: 记录查询结果
  • 工作记忆: 标记任务进行中

  ↓
【第2轮 ReAct】
  Thought: "已获得股价信息，可以回答用户"
  Action: finish()

  ↓
【输出层】
  生成最终答案: "苹果(AAPL)当前股价为 $185.50"
  
  ↓
用户收到答案
```

### 4.3 各组件协作关系

| 组件 | 职责 | 依赖谁 | 被谁依赖 |
|------|------|--------|----------|
| **Temperature** | 控制随机性 | 无（基础配置） | Planning、Reasoning |
| **CoT** | 单路径深度推理 | ReAct 的 Thought | 复杂步骤分析 |
| **ToT/GoT** | 多路径探索 | 规划层决策 | 最终答案选择 |
| **ReAct** | 推理-行动循环 | Tools、Memory | 核心执行引擎 |
| **Memory** | 存储与检索 | RAG 向量库 | 所有需要上下文的模块 |
| **Tools** | 外部交互 | API/数据库 | ReAct 的 Action |
| **Plan-and-Solve** | 全局规划 | 任务分解 | ReAct 的执行序列 |
| **Reflexion** | 错误恢复 | Evaluator | 失败后的重试 |

### 4.4 从简单到复杂的 Agent 演进

你可以根据需求**逐步添加组件**：

```
Level 1: 基础 Agent
  └── LLM + Prompt (Zero-shot)

Level 2: 可控 Agent
  └── + Temperature 控制
  └── + Few-shot 示例

Level 3: 推理 Agent
  └── + CoT (思维链)
  └── + ReAct (工具调用)

Level 4: 记忆 Agent
  └── + 短期记忆 (对话历史)
  └── + RAG (长期记忆)

Level 5: 规划 Agent
  └── + Plan-and-Solve
  └── + ToT/GoT (多方案探索)

Level 6: 自适应 Agent
  └── + Reflexion (自我纠错)
  └── + Multi-Agent (协作)

Level 7: 标准化生态
  └── + MCP (协议标准)
  └── + Skill (技能封装)
```

## 第五章：Agent 核心组件详解

在了解了整体架构之后，现在我们详细展开各个组件的实现细节。

### 5.1 记忆系统（Memory）

| 记忆类型 | 内容 | 实现方式 |
|---------|------|---------|
| 短期记忆 | 对话历史 | 上下文窗口 |
| 长期记忆 | 用户偏好、知识点 | 向量数据库（如 Chroma, Pinecone）|
| 工作记忆 | 当前任务状态 | 变量存储 |

**RAG（检索增强生成）** 是长期记忆的典型应用：
```
用户提问 → 向量化 → 检索相关知识 → 注入 Prompt → LLM 生成
```

### 5.2 工具使用（Tool Use）

通过 **Function Calling** 机制，让 LLM 决定调用什么工具：

```python
tools = [
    {
        "name": "calculator",
        "description": "执行数学计算",
        "parameters": {"expression": "string"}
    },
    {
        "name": "search", 
        "description": "搜索网络信息",
        "parameters": {"query": "string"}
    }
]
```

常见工具类型：
- **搜索**：Google、Bing、内部知识库
- **代码执行**：Python REPL、沙箱环境
- **数据库**：SQL 查询、NoSQL 操作
- **文件操作**：读写文档、代码文件
- **浏览器**：网页浏览、表单填写

### 5.3 工具连接的标准化：MCP 与 Skill

随着 Agent 工具数量的增加，如何**标准化工具接入**和**复用常见功能**成为新的问题。MCP 和 Skill 是两种解决方案。

#### MCP（Model Context Protocol）

**MCP** 是 Anthropic 于 2024 年提出的开放协议，旨在统一 LLM 与外部数据源的连接方式。

**核心思想**：
```
传统方式：每个工具都要写不同的适配代码
MCP 方式：统一协议，一次接入，到处使用
```

**MCP 架构**：

```
┌─────────────────────────────────────────────────────────────┐
│                        AI 客户端                             │
│  ┌──────────┐      ┌──────────────┐                        │
│  │   LLM    │◄────►│  MCP 客户端   │                        │
│  └──────────┘      └──────┬───────┘                        │
└───────────────────────────┼─────────────────────────────────┘
                            │ 标准协议
        ┌───────────────────┼───────────────────┐
        ▼                   ▼                   ▼
┌──────────────┐   ┌──────────────┐   ┌──────────────┐
│  文件系统服务 │   │  数据库服务   │   │  GitHub服务  │
└──────────────┘   └──────────────┘   └──────────────┘
```

**MCP 的优势**：
- **标准化**：统一工具描述、调用方式、错误处理
- **可发现性**：动态获取可用工具列表和能力描述
- **安全性**：明确权限边界，支持用户确认
- **生态复用**：社区共享的 MCP 服务器即插即用

**对比传统 Function Calling**：

| 特性 | 传统 Function Calling | MCP |
|------|---------------------|-----|
| 工具描述 | 硬编码在 Prompt | 动态从服务器获取 |
| 接入方式 | 每个工具单独适配 | 统一协议接入 |
| 社区生态 | 难以共享 | 可共享、可复用 |
| 安全性 | 依赖实现 | 协议级别支持 |

#### Skill（技能封装）

**Skill** 是将常用功能封装成可复用模块的概念，常见于 Claude Code 等工具中。

**什么是 Skill？**
```
Skill = 特定领域的工具集合 + 领域知识 + 最佳实践
```

**Skill 示例**：
- **代码审查 Skill**：包含代码分析工具 + 审查规范 + 自动化检查
- **数据分析 Skill**：包含 SQL 查询 + 可视化工具 + 统计方法
- **文档生成 Skill**：包含模板引擎 + 格式转换 + 多语言支持

**Skill 与单个工具的区别**：

```
┌─────────────────────────────────────────────────────────────┐
│                     Skill（技能封装）                        │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐      │
│  │ 代码审查 Skill │  │ 数据分析 Skill │  │ 文档生成 Skill │      │
│  └───────┬──────┘  └───────┬──────┘  └───────┬──────┘      │
└──────────┼──────────────────┼──────────────────┼───────────┘
           │                  │                  │
           ▼                  ▼                  ▼
    ┌──────────────┐   ┌──────────────┐   ┌──────────────┐
    │   单个工具    │   │   单个工具    │   │   单个工具    │
    │ • 搜索工具   │   │ • 搜索工具   │   │ • 搜索工具   │
    │ • 代码执行   │   │ • API 调用   │   │ • API 调用   │
    └──────────────┘   │ • 代码执行   │   └──────────────┘
                       └──────────────┘
```

**Skill 的价值**：
1. **开箱即用**：封装了完整的工作流，不只是单个工具
2. **最佳实践内置**：包含了该领域的专业知识和经验
3. **可组合**：多个 Skill 可以叠加使用
4. **可共享**：团队内或社区中复用

#### MCP vs Skill：互补关系

| 维度 | MCP | Skill |
|------|-----|-------|
| **定位** | 协议标准 | 功能封装 |
| **解决的问题** | 工具如何连接 | 功能如何组织 |
| **层级** | 底层通信协议 | 上层应用抽象 |
| **关系** | Skill 可以使用 MCP 协议接入工具 | MCP 可以作为 Skill 的基础设施 |

**实际应用**：
```
用户调用"代码审查 Skill"
        ↓
Skill 内部使用 MCP 协议连接：
  - Git MCP 服务器（获取代码）
  - Linter MCP 服务器（静态检查）
  - LLM（分析逻辑问题）
        ↓
整合结果，输出审查报告
```

### 5.4 规划能力集成

实际 Agent 往往是多种方法的组合：

```
用户输入
   ↓
路由判断（简单/复杂）
   ↓
┌──────────────┐  ┌──────────────────┐
│ 简单任务      │  │ 复杂任务          │
│ CoT 直接解决  │  │ Plan-and-Solve   │
│              │  │ + ToT 探索方案     │
│              │  │ + Reflexion 纠错   │
└──────────────┘  └──────────────────┘
   ↓
工具调用（ReAct 模式）
   ↓
结果输出
```

## 总结：从 Prompt 到 Agent 的演进路径

| 阶段 | 能力 | 代表技术 |
|-----|------|---------|
| **Level 1** | 基础对话 | Zero-shot Prompt |
| **Level 2** | 示例学习 | Few-shot, CoT |
| **Level 3** | 单步工具调用 | Function Calling |
| **Level 4** | 多步推理规划 | ReAct, ToT, Plan-and-Solve |
| **Level 5** | 自我改进 | Reflexion |
| **Level 6** | 多 Agent 协作 | Multi-Agent, AutoGen |
| **Level 7** | 标准化生态 | MCP, Skill |

Agent 的发展正在从**单轮响应**走向**自主决策**，从**静态知识**走向**动态交互**。掌握这些基础方法，你就能构建出真正有用的 AI 应用。
